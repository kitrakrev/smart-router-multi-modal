Why Embedding Match, Not Rules?
Rule-Based Routing (Brittle)
if "prove" in query → math if "code" in query → code if "write" in query → ??? Breaks on: "analyze this function's time complexity" Manual rules for every edge case
Embedding Match (Robust)
query embed → cosine sim to decision exemplar centroids "analyze complexity" → code (0.48) NOT math, NOT creative Handles ambiguity naturally Just add exemplars to YAML
Budget-Aware Selection
Not Just "Which Model" — "Cheapest Correct Model" Decision: "This is a code task" require: [code] strategy: cheapest_capable Available models: • GPT-4o [code] ✓ $10.00/1K ← expensive • Claude Sonnet [code] ✓ $3.00/1K • Gemini Flash [code] ✓ $0.40/1K ← PICKED ✓ • Qwen-3B [text] ✗ ← no code capability 3 strategies: cheapest_capable | quality_first | balanced

Technical Decisions
Python over Go/Rust → 3,300 lines vs 654K lines (vLLM-SR) → Same accuracy, faster to iterate → 10ms latency (good enough for routing) MiniLM-L6 over mmBERT-32K → 22M vs 350M params → 10ms vs 243ms per query → 98% vs 66% accuracy (better with good exemplars) No model names in training → Trained on task profiles, not "pick GPT-4" → Add new model tomorrow → works without retraining → Budget layer maps profiles → cheapest capable model Embedding match over classifier → Zero training needed — add exemplars in YAML → 98.1% accuracy without any training data
What We Got Wrong & Learned
✗ RouterArena 98.1% is inflated → gpt-4o-mini has ground truth for all queries → Routing away from it = "wrong" by default → Real accuracy on 3-model overlap: 80.9% ✗ Removing thresholds dropped accuracy to 51% → "What is 2+2" → complex_reasoning (oops) → Need soft thresholds, not hard rules ✗ Fine-tuning on benchmark data = data leakage → Caught it, switched to LMSYS for training ✗ Keyword fallback was silently active → Embeddings weren't loading, nobody noticed → Added loud failure mode — no silent fallbacks
